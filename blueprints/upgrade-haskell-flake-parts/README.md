# upgrade-haskell-flake-parts

An agent-driven seihou **blueprint** that migrates a shinzui Haskell project's
Nix flake, in place, to the thin **flake-parts** structure built on the shared
`haskell-nix-dev` base flake.

It is the vehicle for the fleet-wide migration described in
`docs/plans/1-migrate-shinzui-haskell-projects-to-flake-parts-via-a-seihou-blueprint.md`.
Because every project diverges (custom inputs, overlays, dev-shell tools,
formatter sets, custom checks, package counts), the conversion needs per-project
judgement and a human review — so it is a blueprint, not a deterministic module
migration.

## What it does

Given a project with a monolithic `flake.nix` (pinning `ghc9122`), the blueprint
reshapes it into the target structure:

- `flake.nix` — a thin stub: standard inputs (base flake, followed nixpkgs,
  flake-parts, treefmt-nix, pre-commit-hooks) plus the project's own inputs, and
  a `flake-parts.lib.mkFlake` `outputs` importing the `nix/*` modules and an
  optional `flake.module.nix`.
- `nix/haskell.nix` — the dev shell via the base flake's `mkDevShell` on
  **GHC 9.12.4**.
- `nix/treefmt.nix` — treefmt-nix flake module (`nix fmt` + a treefmt check).
- `nix/pre-commit.nix` — git-hooks.nix flake module (treefmt + custom hooks).
- `flake.module.nix` — **unmanaged**: the package build, the overlay graft, and
  any custom checks. Never regenerated by a template upgrade.

The old top-level `treefmt.nix` is deleted; `nix/haskell-overlay.nix` is kept.

## Guardrails

- The one intended behavior change is the GHC bump **9.12.2 → 9.12.4** (the base
  flake supports only `ghc9124`).
- It never edits Haskell source. If the code does not compile on 9.12.4 it stops
  and reports the exact error.
- It never commits or pushes. A human reviews the diff and the build, then
  commits.

## Usage

From a clean working tree in the project being migrated:

```
seihou agent run upgrade-haskell-flake-parts
```

Then review, regenerate the lock, and build (see the ExecPlan's Concrete Steps),
and commit by hand.

To inspect the rendered system prompt without contacting a provider:

```
seihou agent --debug run upgrade-haskell-flake-parts
```

## Reference files

The `files/` directory mirrors the target tree (`flake.nix`, `nix/haskell.nix`,
`nix/treefmt.nix`, `nix/pre-commit.nix`, `flake.module.nix`) and is mounted
read-only for the agent to adapt.
